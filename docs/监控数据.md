# Worker 监控数据（Elasticsearch）

Worker 每分钟把运行情况写入 **`[product_service]`** 配置的 Elasticsearch（与 monitoring-dashboard 共用同一 ES 集群即可）。

## 索引（所有卖场共用）

| 业务 | 索引名 | 说明 |
|------|--------|------|
| 报价刷新 | `spapi_task_stats_offers` | 全部站点写在同一索引，用 `marketplace` 区分 |
| 目录同步 | `spapi_task_stats_catalog` | 同上 |

旧索引（不再写入）：

- `task_stats_us`、`task_stats_uk` …（按站点拆索引）
- `spapi_item_offers_task_stats`（Offer 监控空壳索引）

## 文档 `_id` 规则

**同一进程、同一站点、同一业务、同一分钟内只有一条汇总文档。**

格式：

```text
{job_type}:{marketplace}:{worker_safe}:pid{pid}:{minute_utc}
```

| 部分 | 含义 | 示例 |
|------|------|------|
| `job_type` | `offers` 或 `catalog` | `offers` |
| `marketplace` | 站点小写 | `us` |
| `worker_safe` | Celery worker 名，`@` → `_at_` | `offers-us-ca-mx_at_gcp-worker-1` |
| `pid` | 操作系统进程号 | `pid88421` |
| `minute_utc` | UTC 分钟，整分 | `2026-07-02T15:04:00Z` |

完整示例：

```text
offers:us:offers-us-ca-mx_at_gcp-worker-1:pid88421:2026-07-02T15:04:00Z
catalog:de:catalog-us-ca-mx_at_gcp-worker-2:pid88422:2026-07-02T15:04:00Z
```

代码位置：`amazon_spapi/amazon/monitoring/task_stats.py` → `build_task_stats_doc_id()`。

## 主要字段（看板 / 查询用）

| 字段 | 含义 |
|------|------|
| `job_type` | `offers` / `catalog` |
| `marketplace` | 站点 |
| `worker` | 节点标识（如 `offers-us-ca-mx@hostname`） |
| `minute` / `time` | 统计分钟（UTC） |
| `task_count` | 这一分钟处理了几批任务 |
| `num_asins` | 请求的 ASIN 总数 |
| `successful_asins` | 成功拿到数据的数量 |
| `failed_asins` | 未拿到或缺失的数量 |
| `api_failed` | Amazon 接口整批失败的次数 |
| `throttle_count` | 本分钟内 Amazon 返回 429（Throttled）的次数（含重试前计数） |
| `task_duration_ms` | 整批任务耗时合计 |
| `spapi_duration_ms` | 调 Amazon 耗时合计 |
| `fetch_gap_ms` / `avg_fetch_gap_ms` | 两批任务间隔（**反映实际吞吐速度**） |
| `avg_task_duration_ms` | 平均每批耗时 |

## 怎么判断节点是否正常

**报价（offers）**

- 查 `spapi_task_stats_offers`，按 `worker` + `marketplace` 过滤
- 最近 5～10 分钟有 `minute` 更新，且 `task_count`、`successful_asins` 在增加 → 正常
- 长时间无新文档，或 `api_failed` / `throttle_count` 持续升高 → 异常

**目录（catalog）**

- 查 `spapi_task_stats_catalog`，同样按 `worker`、`marketplace` 过滤

## monitoring-dashboard 对接

看板仓库：`monitoring-dashboard`。生产环境需配置：

| 环境变量 | 值 |
|----------|-----|
| `SPAPI_ITEM_OFFERS_TASK_STATS_INDEX` | `spapi_task_stats_offers`（代码默认已是此索引） |
| `ES_INDEXES` | 增加 `spapi_task_stats_offers,spapi_task_stats_catalog`（供 **Task Hourly Stats** 采集） |

旧配置（**看不到 c845us 等新 worker**）：

| 原配置 | 问题 |
|--------|------|
| 默认读 `spapi_item_offers_task_stats` | 只有 pl125、t003 等旧 worker 数据 |
| `ES_INDEXES` 仅 `task_stats_us` 等 | collector 不采集 `spapi_task_stats_*` |
| `marketplace.keyword` 聚合 | 新索引 `marketplace` 为 keyword 类型，`.keyword` 子字段为空桶 |

**Task Hourly Stats** 按 `worker` 列展示（如 `c845us-offers@c845us`），需 collector 跑过并写入 PostgreSQL。  
**SP-API item offers** 面板按 **站点** 汇总（CA 的 1h/24h 任务量），不单独显示 worker 名。

### 看板字段：Stats rows 是什么？

`Stats rows` = 在对应时间窗内，ES 索引里该站点有多少条 **分钟汇总文档**（`doc_count`）。

每条文档 = 某 worker 进程 + 某站点 + 某一 UTC 分钟的一条汇总（见上文 `_id` 规则）。  
例如 Last 1h 显示 **56**：过去 1 小时内 CA 约有 56 个「分钟桶」文档被聚合进来（多 worker/多 pid 时同一分钟可能有多条）。

**看板数字不会无限涨**：每列只统计固定窗口（1h / 2h / 24h / 72h），默认 ES 查询跨度 ≥ 72h（`SPAPI_ITEM_OFFERS_STATS_WINDOW_HOURS`）。

**ES 索引本身会持续增长**：`spapi_task_stats_offers` / `spapi_task_stats_catalog` 默认 **保留 7 天**（`SPAPI_TASK_STATS_RETENTION_DAYS`，在 worker 环境 `/etc/conf.d/celery_spapi`）。Worker 启动时通过 Redis 锁每天最多 purge 一次，删除 `minute` 早于保留期的文档。设为 `0` 关闭自动清理。

## 示例查询（Kibana Dev Tools）

```json
GET spapi_task_stats_offers/_search
{
  "size": 5,
  "sort": [{ "minute": "desc" }],
  "query": {
    "bool": {
      "filter": [
        { "term": { "marketplace": "us" } },
        { "range": { "minute": { "gte": "now-1h" } } }
      ]
    }
  }
}
```

```json
GET spapi_task_stats_catalog/_search
{
  "size": 0,
  "query": { "range": { "minute": { "gte": "now-1h" } } },
  "aggs": {
    "by_worker": {
      "terms": { "field": "worker.keyword", "size": 50 },
      "aggs": {
        "by_marketplace": {
          "terms": { "field": "marketplace.keyword", "size": 50 },
          "aggs": {
            "asins_per_min": { "sum": { "field": "successful_asins" } }
          }
        }
      }
    }
  }
}
```
