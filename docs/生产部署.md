# 生产环境部署指南（systemd）

本文说明如何在 Linux 生产服务器上部署 **Amazon 商品数据同步系统** 的执行端，并使用 **systemd** 管理进程。

## Ubuntu 一键部署（推荐）

代码放在 **`~/src/em-workers`**（例如 `/home/sky/src/em-workers`）：

```bash
# 1. 克隆或进入项目
mkdir -p ~/src
git clone https://github.com/AriseshineSky/amazon-spapi-workers.git ~/src/em-workers
cd ~/src/em-workers

# 2. 编辑部署配置
cp deploy/deploy.env.example deploy/deploy.env
vim deploy/deploy.env    # 填 BROKER_URL、OFFERS_*、CATALOG_*、DEPLOY_USER

# 3. 一键部署（需 sudo）
sudo ./deploy/install-ubuntu.sh
```

脚本会自动：安装系统依赖与 uv、同步 Python 包、创建 `~/.amazon_spapi/`、写入 `/etc/conf.d/celery_spapi`、安装并启动 systemd 服务。

| 配置项 | 含义 |
|--------|------|
| `DEPLOY_USER` | 运行服务的用户（如 `sky`） |
| `APP_DIR` | 留空则默认 `~/src/em-workers` |
| `BROKER_URL` | Redis 任务队列地址（必填） |
| `OFFERS_MARKETPLACES` / `CATALOG_MARKETPLACES` | 支持的卖场（逗号分隔，小写） |
| `OFFERS_WORKER_NAME` / `CATALOG_WORKER_NAME` | Celery 节点名（如 `c845us-offers`） |
| `OFFERS_CONCURRENCY` / `CATALOG_CONCURRENCY` | systemd `-c` 并发 |
| `OFFERS_TASK_RATE` / `CATALOG_TASK_RATE` | Celery 任务限速 |

试跑不启动服务：`sudo ./deploy/install-ubuntu.sh --no-start`

修改 worker 参数后：

```bash
sudo vim /etc/conf.d/celery_spapi
sudo systemctl restart amazon-spapi-offers amazon-spapi-catalog
```

调度端（定时提交 ASIN 任务）可放在本机或另一台机器，步骤见文末「调度端部署」。

以下章节为手动部署说明，与脚本等效步骤供参考。

## 部署前准备

### 需要准备的内容

| 项目 | 说明 |
|------|------|
| 服务器 | 建议 Linux（如 Debian/Ubuntu），能访问 Redis、Elasticsearch、Amazon |
| Python | 3.11 或 3.12 |
| uv | Python 包管理工具，[安装说明](https://github.com/astral-sh/uv) |
| Redis | 任务队列，调度端与执行端共用同一地址 |
| 配置文件 | Amazon 凭证、Elasticsearch 连接等（见下文） |
| 系统用户 | 建议专用账号运行，不要用 root |

### 建议的服务划分

按**站点**和**业务线**拆成多个 systemd 服务，互不影响、便于单独重启：

| 服务名（示例） | 负责的业务 | 监听队列（示例） |
|----------------|------------|------------------|
| `amazon-spapi-offers` | 多站报价刷新 | `SpapiItemOffersUpdate_US,SpapiItemOffersUpdate_CA,…` |
| `amazon-spapi-catalog` | 多站目录同步 | `SpapiCatalogItemsUpdate_US,…` |

报价刷新和目录同步**建议分开进程**：Amazon 对两类接口的频率限制不同，分开更易控速。

## 第一步：安装代码与依赖

```bash
# 以专用用户登录，例如 deploy
sudo useradd -m -s /bin/bash amazon-spapi   # 若尚未创建
sudo su - amazon-spapi

# 安装 uv（若系统尚未安装）
curl -LsSf https://astral.sh/uv/install.sh | sh

# 部署目录（可按规范调整）
export APP_HOME=/opt/amazon-spapi-workers
git clone git@github.com:VG-IT/em-workers.git "$APP_HOME"
cd "$APP_HOME"
uv sync --frozen
```

确认安装成功：

```bash
cd /opt/amazon-spapi-workers
BROKER_URL=redis://127.0.0.1:6379/2 uv run python -c "from amazon_spapi.worker import app; print('ok')"
```

## 第二步：配置文件

### 1. 业务配置（Amazon + Elasticsearch）

```bash
mkdir -p ~/.amazon_spapi/logs
cp /opt/amazon-spapi-workers/config.ini.sample ~/.amazon_spapi/config.ini
chmod 600 ~/.amazon_spapi/config.ini
vim ~/.amazon_spapi/config.ini
```

需填写的主要段落：

| 配置段 | 用途 |
|--------|------|
| `[spapi]` | Amazon 卖家接口凭证（LWA、AWS 密钥等） |
| `[offer_service]` | 存报价的 Elasticsearch |
| `[product_service]` | 存商品/统计的 Elasticsearch |
| `[telegram]` / `[sentry]` | 可选告警 |

### Worker 参数（`/etc/conf.d/celery_spapi`）

**队列、并发、worker 名、任务限速** 全部在环境文件中配置，systemd 单元通过 `${OFFERS_QUEUES}` 等变量展开。

模板：`deploy/conf.d/celery_spapi.example`

| 服务 | 监听队列 |
|------|----------|
| `amazon-spapi-offers` | `OFFERS_QUEUES`（或由 `OFFERS_MARKETPLACES` 自动生成） |
| `amazon-spapi-catalog` | `CATALOG_QUEUES`（或由 `CATALOG_MARKETPLACES` 自动生成） |

示例（美加墨三站，worker 名 `c845us-*`）：

```bash
OFFERS_WORKER_NAME=c845us-offers
OFFERS_CONCURRENCY=1
OFFERS_MARKETPLACES=us,ca,mx
OFFERS_QUEUES=SpapiItemOffersUpdate_US,SpapiItemOffersUpdate_CA,SpapiItemOffersUpdate_MX
OFFERS_TASK_RATE=16/m

CATALOG_WORKER_NAME=c845us-catalog
CATALOG_CONCURRENCY=1
CATALOG_MARKETPLACES=us,ca,mx
CATALOG_QUEUES=SpapiCatalogItemsUpdate_US,SpapiCatalogItemsUpdate_CA,SpapiCatalogItemsUpdate_MX
CATALOG_TASK_RATE=1/s
```

| 文件 | 用途 |
|------|------|
| **`/etc/conf.d/celery_spapi`** | `BROKER_URL`、`AMAZON_SPAPI_CONFIG_PATH`、全部 `OFFERS_*` / `CATALOG_*` |

### Worker 节点名（便于排查哪个程序没跑）

每个 Celery 进程通过 `-n ${OFFERS_WORKER_NAME}@主机` 注册。该名字出现在 journal、`celery inspect ping`、ES 监控 `worker` 字段。检查是否都在线：

```bash
BROKER_URL='...' uv run celery -A amazon_spapi.worker inspect ping
# 或
amazon-spapi-check --with-workers
```

缺哪个名字，就说明对应 systemd 服务（`amazon-spapi-offers` / `amazon-spapi-catalog`）未运行或配置不一致。

### 并发与 SP-API 速率（如何不超限）

系统里有 **三层** 限速，需一起算：

| 层级 | 配置位置 | 作用 |
|------|----------|------|
| **入队速率** | 调度脚本 `-q`（catalog 入队） | 控制任务进入 Redis 的速度 |
| **Worker 并发** | `OFFERS_CONCURRENCY` / `CATALOG_CONCURRENCY` → systemd `-c` | 同时跑任务的 **子进程数** |
| **任务 rate_limit** | `OFFERS_TASK_RATE` / `CATALOG_TASK_RATE` | **每个子进程** 每分钟/秒最多执行几次该任务 |

**估算 SP-API 调用次数：**

```
实际 API 请求/分钟 ≈ 并发数 × 每进程任务速率 × 60
```

（`8/m` 表示每进程每分钟 8 个任务；`1/s` 表示每进程每秒 1 个任务。）

每个任务 = **1 次 batch 请求**，最多 **20 个 ASIN**（`GetItemOffersBatch` / `searchCatalogItems`）。

**示例 — 报价：**

- `offers_concurrency = 2`，`offers_task_rate = 8/m`
- 总任务 ≈ 2 × 8 = **16 任务/分钟** → 最多 **16 次** GetItemOffersBatch/分钟（全站点共用同一任务名，Celery 会在进程内合并限速）

**示例 — 目录：**

- `catalog_concurrency = 1`，`catalog_task_rate = 1/s`
- 总任务 ≈ 1 × 60 = **60 任务/分钟** → 若 Amazon 账号 quota 更低，应降低 `catalog_task_rate` 或 `catalog_concurrency`

**429 / Throttled：** 若仍触发 Amazon 限流，任务会 `Reject` 并重新入队；长期超限请调低上述参数。Seller Central → Apps & Services → Develop Apps 可查看各 API 的 **Rate / Burst** quota。

**建议：**

1. 报价与目录 **分开进程**（默认已是两个 systemd 单元），互不影响。
2. 多站点可共用一个 offers worker（队列用逗号连接）；限速按 **任务类型** 算是全站点合计，有利于总 cap。
3. 先保守（如 offers `4/m`、catalog `0.5/s`），观察 ES 监控索引 `spapi_task_stats_*` 与日志，再逐步提高。

修改 `/etc/conf.d/celery_spapi` 后需 **重启 worker** 才会加载新的 `rate_limit`（settings 在进程启动时读取环境）。

### 2. 运行环境（Redis 等敏感信息）

**不要把 Redis 密码写在 systemd 单元文件里**（易被 `ps`、日志看到）。使用 `/etc/conf.d/` 环境文件：

```bash
sudo cp /opt/amazon-spapi-workers/deploy/conf.d/celery_spapi.example \
  /etc/conf.d/celery_spapi
sudo chmod 640 /etc/conf.d/celery_spapi
sudo chown root:Admin /etc/conf.d/celery_spapi
sudo vim /etc/conf.d/celery_spapi
```

每个文件示例内容见 `deploy/conf.d/celery_spapi.example`（含 `BROKER_URL` 与全部 worker 变量）。

## 第三步：安装 systemd 服务

**推荐：**

```bash
sudo cp deploy/conf.d/celery_spapi.example /etc/conf.d/celery_spapi
sudo vim /etc/conf.d/celery_spapi   # 填 BROKER_URL、OFFERS_*、CATALOG_*

sudo ./deploy/install-systemd.sh \
  --user Admin \
  --app-dir ~/src/em-workers

sudo systemctl enable --now amazon-spapi-offers amazon-spapi-catalog
```

单元模板 `deploy/systemd/*.service` 已固定，**无需**按 config.ini 生成；队列/并发/名字由 `EnvironmentFile` 提供。

加载并启动：

```bash
sudo systemctl daemon-reload
sudo systemctl enable amazon-spapi-offers amazon-spapi-catalog
sudo systemctl start amazon-spapi-offers amazon-spapi-catalog
sudo systemctl status amazon-spapi-offers
```

查看运行日志：

```bash
sudo journalctl -u amazon-spapi-offers -u amazon-spapi-catalog -f
```

## 日志目录与 logrotate

### 日志在哪里

| 类型 | 路径 | 说明 |
|------|------|------|
| **调度 / 入队脚本** | `~/.amazon_spapi/logs/` | 如 `spapi_update_item_offers_task_sender.log`、`schedule_stale_offers.log` |
| **cron 重定向** | `~/.amazon_spapi/logs/cron.log` | crontab 里 `>> …/cron.log` 时使用 |
| **Celery worker** | **journald**（无单独文件） | `journalctl -u amazon-spapi-offers -u amazon-spapi-catalog` |

VPS **c845us** 上 Admin 用户的路径：

```text
/home/Admin/.amazon_spapi/logs/
```

入队脚本内置 `RotatingFileHandler`（单文件约 20MB × 5 备份），但 **cron.log** 和 **journald** 仍需系统级限制。

### 安装 logrotate（防止占满硬盘）

一键部署已包含；手动安装：

```bash
cd ~/src/em-workers
sudo ./deploy/install-logrotate.sh --user Admin
```

会安装：

| 文件 | 作用 |
|------|------|
| `/etc/logrotate.d/amazon-spapi` | 轮转 `~/.amazon_spapi/logs/*.log`、`cron.log`（保留 14 天/8 周，gzip） |
| `/etc/systemd/journald.conf.d/amazon-spapi-size.conf` | journal 总量上限 **512MB**，最长 **14 天** |

测试 logrotate：

```bash
sudo logrotate -f /etc/logrotate.d/amazon-spapi
```

查看 journal 占用：

```bash
journalctl --disk-usage
```

## systemd 单元文件说明

两个服务的逐步代码执行路径（从 systemd 启动到 SP-API 调用、写 ES）见 **[systemd 执行路径](./systemd执行路径.md)**。

模板位置：`deploy/systemd/*.service`

| 配置项 | 含义 |
|--------|------|
| `User` | 运行服务的系统用户 |
| `WorkingDirectory` | 项目根目录（含 `pyproject.toml`） |
| `EnvironmentFile` | 公用 → `/etc/conf.d/celery_spapi` |
| `ExecStart` | 启动执行端，`-Q` 指定监听的业务队列 |
| `-c 2` | 同时处理 2 条任务（可按 CPU 与 Amazon 限流调整） |
| `Restart=always` | 崩溃后自动拉起 |

### 报价刷新（US 站）示例

```ini
[Unit]
Description=Amazon SP-API — US 站报价刷新
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=amazon-spapi
Group=amazon-spapi
WorkingDirectory=/opt/amazon-spapi-workers
EnvironmentFile=/etc/conf.d/celery_spapi

ExecStart=/home/amazon-spapi/.local/bin/uv run celery -A amazon_spapi.worker worker \
  -l info \
  -c 2 \
  -Q SpapiItemOffersUpdate_US

Restart=always
RestartSec=15
KillSignal=SIGTERM
TimeoutStopSec=120

[Install]
WantedBy=multi-user.target
```

### 目录同步（US 站）示例

将 `-Q` 改为 `SpapiCatalogItemsUpdate_US`，并发 `-c 1` 通常即可（Amazon 目录接口更严）。

### 多站点

复制 service 文件，改服务名和 `-Q` 后缀即可，例如：

- `SpapiItemOffersUpdate_UK`
- `SpapiItemOffersUpdate_DE`
- `SpapiCatalogItemsUpdate_AE`

## 第四步：验证上线

### 1. 确认进程在跑

```bash
sudo systemctl is-active amazon-spapi-offers amazon-spapi-catalog
```

### 2. 确认已注册任务类型

```bash
cd /opt/amazon-spapi-workers
set -a && source /etc/conf.d/celery_spapi && set +a
uv run celery -A amazon_spapi.worker inspect registered
```

应能看到：

- `amazon_spapi.jobs.refresh_offers.refresh_offers`
- `amazon_spapi.jobs.refresh_catalog.refresh_catalog`
- `amazon_spapi.jobs.fetch_products.fetch_products`

### 3. 提交一条小批量试跑

在**调度端**（可与生产机不同）：

```bash
export BROKER_URL='redis://:密码@生产Redis地址:6379/2'
cd /path/to/em-workers
echo "B012345678" > /tmp/test_asin.txt   # 换成真实 ASIN
uv run schedule-stale-offers -m us -q 1 -t 36 -f -p 9 /tmp/test_asin.txt
```

回到生产机看日志是否有处理记录，并在 Elasticsearch 确认报价时间是否更新。

### 4. 看队列是否被消费

```bash
redis-cli -n 2 LLEN SpapiItemOffersUpdate_US
```

数字应下降或保持低位；若只增不减，检查 `-Q` 是否与提交任务的站点一致。

## 日常运维

### 常用命令

```bash
# 启动 / 停止 / 重启
sudo systemctl start amazon-spapi-offers amazon-spapi-catalog

sudo systemctl stop amazon-spapi-offers amazon-spapi-catalog

sudo systemctl restart amazon-spapi-offers amazon-spapi-catalog

sudo systemctl enable amazon-spapi-offers amazon-spapi-catalog

sudo journalctl -u amazon-spapi-offers -u amazon-spapi-catalog -n 200 --no-pager
```

### 如何判断是否正常

| 现象 | 可能原因 | 建议 |
|------|----------|------|
| 队列长度持续升高 | 执行端挂了或太慢 | `systemctl status`、加 `-c` 或加机器 |
| 任务提交无反应 | Redis 地址不一致 | 对比调度端与 `/etc/conf.d/celery_spapi` |
| 日志出现 Forbidden | Amazon 凭证或权限问题 | 查 config.ini；系统会 Telegram 告警 |
| 服务频繁重启 | 配置错误或依赖不可达 | `journalctl -xe` 查具体报错 |

### 告警

- **Telegram**：Amazon 权限被拒（Forbidden）时通知运维  
- **Sentry**：在 `config.ini` 配置 `[sentry]` 后，不可恢复错误会上报

## 版本升级（推荐顺序）

避免调度端和新执行端行为不一致：

```
1. 调度端：git pull && uv sync --frozen
2. 小批量试跑 1～2 条任务
3. 生产执行端：git pull && uv sync --frozen
4. 滚动重启：先停 US 报价 → 启动 → 确认正常 → 再其他站点
```

```bash
cd /opt/amazon-spapi-workers
git pull
uv sync --frozen
sudo systemctl restart amazon-spapi-offers amazon-spapi-catalog
```

### 升级时不要随意改动的约定

| 项目 | 示例 | 原因 |
|------|------|------|
| 队列名称 | `SpapiItemOffersUpdate_US` | 调度端与执行端必须一致 |
| Redis DB 号 | `/6379/2` | 否则任务进错库 |
| 任务参数顺序 | 站点 + ASIN 列表 + 条件 | 旧消息仍在队列里 |

业务规则（过期时间、过滤条件）可以只升级调度端；**改执行端业务逻辑时，执行端必须同步部署**。

## 调度端部署（cron）

调度端负责「决定刷哪些 ASIN」，通常跑在办公机或内网 cron，**不需要 systemd 保活**，定时跑即可。

### 环境文件

```bash
# /home/amazon-spapi/.amazon-spapi/cron.env
BROKER_URL=redis://:密码@生产Redis:6379/2
AMAZON_SPAPI_CONFIG_PATH=/home/amazon-spapi/.amazon_spapi/config.ini
```

### cron 示例：每天凌晨刷 US 过期报价

```cron
0 2 * * * set -a && . /home/amazon-spapi/.amazon-spapi/cron.env && set +a && \
  cd /opt/amazon-spapi-workers && \
  /home/amazon-spapi/.local/bin/uv run schedule-stale-offers \
  -m us -q 20 -t 36 -p 5 /data/asins_us.txt \
  >> /home/amazon-spapi/.amazon_spapi/logs/cron.log 2>&1
```

### 其他调度方式

| 方式 | 适用 |
|------|------|
| `schedule-stale-offers` | 从 ASIN 文件 + 过期过滤（最常用） |
| `enqueue_offers_from_es` | 从 Elasticsearch 索引批量提交报价任务 |
| `enqueue_catalog_from_asins` | 从 ASIN 文件提交目录任务 |

模块路径见 `amazon_spapi/commands/amazon/`。

## 任务与队列对照表

| 业务 | 任务名 | 队列格式 | 执行频率限制（内置） |
|------|--------|----------|----------------------|
| 报价刷新 | `refresh_offers` | `SpapiItemOffersUpdate_{站点}` | 约 8 次/分钟/进程 |
| 目录同步 | `refresh_catalog` | `SpapiCatalogItemsUpdate_{站点}` | 约 1 次/秒/进程 |
| 组合拉取 | `fetch_products` | Celery 默认队列 | 约 6 次/分钟/进程 |

`{站点}` 为大写：`US`、`UK`、`DE`、`AE` 等。
