# Amazon 商品数据同步系统 — 业务说明

## 这个系统是做什么的

EveryMarket 需要持续掌握 Amazon 上各站点的**最新报价**和**商品目录信息**（标题、图片、属性等），才能做定价、上架和库存判断。

本系统负责这件事的后台部分：

1. **收到一批要更新的商品编号（ASIN）**
2. **向 Amazon 批量查询**最新报价或目录
3. **把结果写回我们的搜索库（Elasticsearch）**，供网站和其他系统使用

整个过程在后台自动跑，不阻塞前台业务。

---

## 解决什么业务问题

| 业务需求 | 系统怎么做 |
|----------|------------|
| 报价过期，定价不准 | 定期找出「超过 N 小时没更新」的商品，重新向 Amazon 拉报价 |
| 新商品缺少目录信息 | 批量拉取标题、图片、类目等，写入商品库 |
| 站点多（US、UK、DE…） | 按站点分开排队、分开部署，互不影响 |
| 紧急补刷 vs 日常全量 | 任务可分「优先处理」和「慢慢排队」，避免大批量拖慢紧急任务 |

## 系统里两类角色

可以把它想成**餐厅**：有人点菜，有人做菜。

```
┌─────────────────┐         ┌──────────────┐         ┌─────────────────┐
│  调度端          │  下单   │  任务队列     │  取单   │  执行端          │
│  （本机 / 定时） │ ──────▶ │  （Redis）   │ ◀────── │  （生产服务器）  │
└─────────────────┘         └──────────────┘         └─────────────────┘
     决定「刷哪些 ASIN」           暂存待办任务              调 Amazon、写 ES
```

| 角色 | 谁在用 | 做什么 |
|------|--------|--------|
| **调度端** | 运营脚本、定时任务、同事手动跑 | 从文件或商品库选出 ASIN，提交「待刷新」任务 |
| **执行端** | 生产服务器上长期运行的进程 | 从队列取任务，访问 Amazon，更新 Elasticsearch |

调度端和执行端**可以不在同一台机器**，只要连同一个任务队列即可。

---

## 三条业务线

### 1. 报价刷新（最常用）

- **业务含义**：某站点某批商品的「当前卖家报价」更新到最新
- **典型场景**：每天凌晨把 US 站过期超过 36 小时的 ASIN 重新刷一遍
- **Amazon 侧**：批量查每个 ASIN 的卖家报价
- **写入位置**：报价相关的 Elasticsearch 索引（如 `lowest_offer_listings_us_new`）
- **生产队列名示例**：`SpapiItemOffersUpdate_US`（UK 则为 `_UK`，以此类推）

### 2. 目录同步

- **业务含义**：商品的标题、品牌、图片、属性等「目录信息」更新
- **典型场景**：新 ASIN 入库后补全商品详情；目录长期未更新时批量补刷
- **Amazon 侧**：按 ASIN 搜索目录条目
- **生产队列名示例**：`SpapiCatalogItemsUpdate_US`

### 3. 商品组合拉取

- **业务含义**：一次任务同时拉「报价 + 目录」，写入商品 API 索引
- **典型场景**：商品详情页需要报价和目录一起展示时使用
- **使用频率**：相对前两条较少，走默认任务队列

每条线**一次最多处理 20 个 ASIN**（Amazon 接口限制）。

---

## 一次报价刷新的完整流程（业务视角）

1. **准备 ASIN 列表**  
   例如从运营导出的 `asins_us.txt`，或从 Elasticsearch 按条件筛出。

2. **过滤「还需要刷」的商品**（可选）  
   若报价在 Elasticsearch 里仍新鲜（例如 36 小时内更新过），则跳过，省 Amazon 调用次数。

3. **提交任务**  
   每 20 个 ASIN 打包成一条任务，放进对应站点的队列。

4. **执行端处理**  
   - 向 Amazon 批量要报价  
   - 转换为我们内部格式  
   - 写入 Elasticsearch  
   - 若某 ASIN 已有报价，可从「无报价待处理」列表中移除

5. **下游使用**  
   定价、listing、运营后台从 Elasticsearch 读最新数据。

## 任务优先级（业务含义）

| 优先级 | 适合什么情况 |
|--------|--------------|
| **最高（9）** | 手工紧急补刷、故障恢复、少量关键 SKU |
| **普通（5）** | 日常定时任务 |
| **最低（0）** | 全量回填、历史数据补刷，不着急 |

优先级高的大批任务会先被处理；低优先级大批量任务可以单独排队，避免拖慢日常更新。

Redis 子队列与 `-p` 一致（**后缀数字越大越优先**）：

| `-p` | Redis 列表（示例 US 报价队列） |
|------|--------------------------------|
| 9 | `SpapiItemOffersUpdate_US:9` |
| 5 | `SpapiItemOffersUpdate_US:5` |
| 0 | `SpapiItemOffersUpdate_US`（无后缀，最低） |

## 项目文件夹在业务上代表什么

不用记技术名词，按「谁负责什么业务」理解即可：

| 文件夹 | 业务上是什么 |
|--------|--------------|
| `amazon/` | **核心业务规则**：怎么刷报价、怎么刷目录、哪些 ASIN 算过期、刷完怎么清理无效记录 |
| `spapi/` | **和 Amazon 打交道**：登录凭证、按站点调用、把 Amazon 返回的数据转成我们用的格式 |
| `services/` | **和我们的 Elasticsearch 打交道**：保存报价、查旧报价、写商品文档 |
| `jobs/` | **执行端接到任务后怎么干**：把「队列里的一条任务」接到上面的业务规则上 |
| `commands/` | **调度端用的命令行工具**：从文件或 ES 选 ASIN、提交任务 |
| `scheduling/` | **排队策略**：优先级、防重复提交、控制每秒提交多少任务 |
| `worker/` | **执行端程序入口**：生产服务器上 `systemd` 启动的就是它 |
| `platform/` | **环境和连接配置**：读配置文件、连 Amazon、连 ES、异常告警 |
| `config/` | **配置文件路径、环境变量** 的约定 |

依赖方向（便于维护，业务上可理解为）：

- **业务规则**（`amazon/`）不依赖「是不是在服务器上跑」  
- **和外界通信**（Amazon、ES）放在外围  
- **命令行和后台进程**只是两种不同的「入口」，背后同一套业务规则

## 谁日常怎么用

| 角色 | 典型操作 |
|------|----------|
| **运维** | 在生产机用 systemd 保活执行端；看队列是否积压；升级代码后滚动重启 |
| **运营 / 数据** | 准备 ASIN 文件；跑「过期报价调度」；必要时 `-f` 强制全量刷 |
| **开发** | 改业务规则、对接新站点、调整过期时间或过滤条件 |

常用调度命令示例（报价）：

```bash
# 只刷 ES 里已超过 36 小时未更新的 US 站报价
uv run schedule-stale-offers -m us -q 20 -t 36 asins_us.txt
```

参数含义：

| 参数 | 业务含义 |
|------|----------|
| `-m us` | 美国站 |
| `-q 20` | 每秒向队列提交约 20 条任务（每条 20 个 ASIN） |
| `-t 36` | 超过 36 小时未更新的才刷 |
| `-f` | 不管是否过期，全部提交 |
| `-p 9` | 最高优先级（紧急） |

## 和外部系统的关系

```
                    ┌─────────────┐
  ASIN 列表 / ES ──▶│  本系统      │──▶ Elasticsearch（报价 / 商品）
                    │             │
                    └──────┬──────┘
                           │
                           ▼
                    Amazon SP-API
                    （官方卖家接口）
```

- **输入**：ASIN 列表、Elasticsearch 里已有数据（判断是否过期）  
- **输出**：更新后的报价 / 目录，供 EveryMarket 定价与商品展示  
- **限制**：受 Amazon 接口频率限制；Forbidden 时会告警并暂停该 worker
